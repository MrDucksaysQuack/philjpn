# Admin 기능 재분류 계획 (UX & 인지 패턴 기반)

## 📊 현재 구조 분석

### 현재 Admin 메뉴 구조 (12개)
1. **사용자 관리** (`/admin/users`)
2. **시험 관리** (`/admin/exams`)
3. **시험 템플릿** (`/admin/templates`)
4. **문제 풀 관리** (`/admin/question-pools`)
5. **시험 결과 모니터링** (`/admin/exam-results`)
6. **라이선스 키 관리** (`/admin/license-keys`)
7. **실시간 모니터링** (`/admin/monitoring`)
8. **사이트 설정** (`/admin/settings`)
9. **카테고리 관리** (`/admin/categories`)
10. **배지 관리** (`/admin/badges`)
11. **문제 은행 관리** (`/admin/question-banks`)
12. **문제 관리** (`/admin/questions`)

## 🎯 인지 패턴 기반 재분류 원칙

### 1. 작업 흐름 기반 그룹화 (Workflow-based)
- **시험 생성 워크플로우**: 템플릿 → 문제 풀 → 시험 생성
- **콘텐츠 관리 워크플로우**: 카테고리 → 문제 은행 → 문제 관리
- **운영 관리 워크플로우**: 사용자 → 라이선스 → 모니터링

### 2. 빈도 기반 그룹화 (Frequency-based)
- **자주 사용**: 시험 관리, 사용자 관리, 결과 모니터링
- **가끔 사용**: 템플릿, 문제 풀, 문제 은행
- **설정**: 사이트 설정, 카테고리, 배지

### 3. 관련성 기반 그룹화 (Relatedness-based)
- **콘텐츠 관련**: 시험, 문제, 템플릿, 카테고리
- **사용자 관련**: 사용자, 라이선스 키
- **분석 관련**: 결과, 모니터링
- **시스템 관련**: 설정, 배지

## 🗂️ 제안된 재분류 구조

### 📝 1. 콘텐츠 관리 (Content Management)
**목적**: 시험과 문제 콘텐츠를 생성하고 관리
**인지 부하**: 낮음 (명확한 그룹화)

```
📁 콘텐츠 관리
├── 📄 시험 관리 (/admin/exams)
│   ├── 시험 목록
│   ├── 시험 생성
│   └── 시험 수정/삭제
├── 📄 문제 관리 (/admin/questions)
│   ├── 전체 문제 조회
│   ├── 문제 검색
│   └── 문제 편집
├── 📄 문제 은행 (/admin/question-banks)
│   ├── 문제 은행 목록
│   └── 문제 은행 관리
├── 📄 문제 풀 (/admin/question-pools)
│   ├── 문제 풀 목록
│   └── 태그/난이도별 그룹화
└── 📄 시험 템플릿 (/admin/templates)
    ├── 템플릿 목록
    └── 템플릿 생성/수정
```

**이유**:
- 시험 생성 시 필요한 모든 기능을 한 곳에 모음
- 작업 흐름: 템플릿 선택 → 문제 풀 선택 → 시험 생성
- 인지적 일관성: 모두 "콘텐츠"라는 공통 개념

---

### 👥 2. 사용자 및 접근 관리 (User & Access Management)
**목적**: 사용자와 라이선스 키 관리
**인지 부하**: 낮음 (명확한 그룹화)

```
📁 사용자 및 접근 관리
├── 📄 사용자 관리 (/admin/users)
│   ├── 사용자 목록
│   ├── 사용자 상세
│   └── 사용자 권한 관리
└── 📄 라이선스 키 (/admin/license-keys)
    ├── 키 목록
    ├── 키 생성 (단일/배치)
    └── 키 통계
```

**이유**:
- 사용자와 접근 권한이 밀접하게 연관
- 라이선스 키는 사용자 접근 제어의 일부
- 작업 흐름: 사용자 생성 → 라이선스 키 발급

---

### 📊 3. 분석 및 모니터링 (Analytics & Monitoring)
**목적**: 시험 결과와 실시간 활동 모니터링
**인지 부하**: 낮음 (명확한 그룹화)

```
📁 분석 및 모니터링
├── 📄 시험 결과 (/admin/exam-results)
│   ├── 전체 결과 조회
│   ├── 결과 분석
│   └── 통계 대시보드
└── 📄 실시간 모니터링 (/admin/monitoring)
    ├── 진행 중인 시험
    ├── 실시간 활동
    └── 시스템 상태
```

**이유**:
- 둘 다 "모니터링"이라는 공통 목적
- 결과는 과거 데이터, 모니터링은 현재 데이터
- 시간적 연속성: 과거 → 현재

---

### ⚙️ 4. 시스템 설정 (System Settings)
**목적**: 플랫폼 전반의 설정 관리
**인지 부하**: 낮음 (명확한 그룹화)

```
📁 시스템 설정
├── 📄 사이트 설정 (/admin/settings)
│   ├── 기본 정보
│   ├── 디자인/테마
│   └── 콘텐츠 관리
├── 📄 카테고리 관리 (/admin/categories)
│   ├── 카테고리 목록
│   └── 서브카테고리 관리
└── 📄 배지 관리 (/admin/badges)
    ├── 배지 목록
    └── 배지 생성/수정
```

**이유**:
- 모두 "설정"이라는 공통 개념
- 자주 변경되지 않는 기능들
- 플랫폼 전반에 영향을 미치는 설정

## 🎯 인지 패턴 고려사항

### 1. 작업 흐름 최적화
**시험 생성 워크플로우**:
```
1. 카테고리 선택 (시스템 설정)
2. 템플릿 선택 또는 직접 생성 (콘텐츠 관리)
3. 문제 풀 선택 (콘텐츠 관리)
4. 시험 생성 (콘텐츠 관리)
5. 라이선스 키 발급 (사용자 및 접근)
```

**개선된 워크플로우**:
- 콘텐츠 관리 섹션 내에서 모든 단계 완료 가능
- 관련 기능들이 인접하게 배치

### 2. 인지 부하 감소
- **그룹 수**: 12개 → 4개 (67% 감소)
- **각 그룹 내 항목**: 평균 3개 (인지 부하 적정)
- **계층 깊이**: 2단계 (최적)

### 3. 발견 가능성 향상
- **관련 기능 그룹화**: 비슷한 기능을 함께 배치
- **명확한 라벨링**: 그룹명이 기능을 명확히 설명
- **시각적 구분**: 아이콘과 색상으로 그룹 구분

### 4. 학습 곡선 최적화
- **초보자**: 그룹명만 보고도 기능 위치 파악 가능
- **숙련자**: 작업 흐름에 따라 빠르게 접근 가능
